\title{\textbf{What Does a Robot Policy ``Look At''?}\\
\large Visualising RDT-1B's Attention with Blur Integrated Gradients}
\author{Brandon Rodriguez}
\date{\today}

\begin{document}
\maketitle

\begin{abstract}
When a robot arm picks up a cube, which part of the camera image is it actually
``paying attention to''? This report answers that question for \textbf{RDT-1B},
a large robot manipulation policy, running inside the \textbf{ManiSkill3} simulation.
We use two techniques from explainable AI---\textbf{Blur Integrated Gradients (BlurIG)}
and \textbf{Integrated Gradients (IG)}---to produce coloured heatmaps that highlight
which image patches and which words in the task instruction drove each robot
joint's actions at any given moment. We collected real successful rollout episodes
across five tasks (PickCube, StackCube, PushCube, PegInsertionSide, PlugCharger),
ran attribution at ten frames per episode, and produced spatial heatmaps, per-joint
breakdowns, and word-level attribution maps. We also document and fix two bugs
found along the way: a random noise problem that was corrupting gradient estimates,
and an attention mask bug that made meaningless ``padding'' tokens appear important.
\end{abstract}

\section{Introduction}
% ─────────────────────────────────────────────────────────────────────────────

Modern robot policies are trained end-to-end on large datasets, meaning they
learn entirely from examples rather than being hand-coded with rules. This makes
them powerful---but it also makes them hard to understand. If a robot fails to
insert a peg, we cannot easily tell \emph{why}: was it looking at the wrong part
of the image? Did it misinterpret the language instruction? Did one joint ignore
the visual input entirely?

\textbf{Attribution methods} try to answer these questions. Given a neural network
and an input, an attribution method assigns a score to each input feature (for
example, each pixel or each word) that reflects how much that feature influenced
the network's output. High-attribution regions are the things the model ``cared
about'' when making its decision.

This project applies attribution methods to \textbf{RDT-1B}~\cite{Liu2024RDT1B},
a 1-billion-parameter robot policy trained to follow language instructions while
looking at camera images. The policy is tested inside \textbf{ManiSkill3}~\cite{Chen2024ManiSkill3},
a robot simulation environment. We ask two questions:

\begin{enumerate}
    \item \textbf{Image attribution:} Which parts of the camera image most
          influenced the robot's joint movements at each moment in time?
    \item \textbf{Language attribution:} Which words in the task instruction
          (e.g.\ ``pick up a \emph{red} cube and stack it on top of a
          \emph{green} cube'') most influenced each individual robot joint?
\end{enumerate}

\section{Background}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{RDT-1B: The Robot Policy}

RDT-1B~\cite{Liu2024RDT1B} is a robot manipulation policy with about one billion
parameters. Think of it as a very large neural network that takes in:
\begin{itemize}
    \item A text instruction (e.g.\ ``stack the cubes'')
    \item Camera images of the scene
    \item The robot's current joint positions (called \emph{proprioception})
\end{itemize}
and outputs a sequence of 64 future joint position targets (a ``plan'' for
the next few seconds).

RDT is a \emph{diffusion model}~\cite{Ho2020DDPM,Chi2023DiffusionPolicy},
meaning it generates its action plan the same way an image diffusion model
generates a picture: it starts from random noise and gradually refines it
into a coherent trajectory through a denoising process. During inference
we use just 5 denoising steps for speed.

The model was originally pre-trained on large robot datasets and then
\emph{fine-tuned} specifically on ManiSkill tasks using a checkpoint from
HuggingFace (\texttt{robotics-diffusion-transformer/maniskill-model}).

\paragraph{How the robot's state is represented.}
The Franka Panda arm has 7 joints plus a gripper, giving 8 numbers that describe
where the arm is at any moment. RDT places these 8 numbers into a larger
128-dimensional vector (most entries are zero for this robot) so that many
different robot types can share the same model architecture.

\subsection{ManiSkill3: The Simulation}

ManiSkill3~\cite{Chen2024ManiSkill3} (and its predecessor ManiSkill2~\cite{Gu2023ManiSkill2})
is a physics simulation of robot manipulation tasks. We use five tasks:

\begin{table}[h]
\centering
\caption{The five tasks studied in this project.}
\label{tab:tasks}
\begin{tabular}{ll}
\toprule
\textbf{Task} & \textbf{What the robot must do} \\
\midrule
PickCube         & Pick up a red cube and move it to a marked goal spot \\
StackCube        & Pick up the red cube and place it on top of the green cube \\
PushCube         & Push a cube across the table to a goal region \\
PegInsertionSide & Pick up a two-coloured peg and insert the orange end into a slot \\
PlugCharger      & Insert a charger-shaped piece into the correct slot on a board \\
\bottomrule
\end{tabular}
\end{table}

The simulation runs at 25 frames per second. The robot is controlled by
telling it where each joint should be (called \texttt{pd\_joint\_pos} control),
and it has at most 400 steps to complete a task.

\subsection{Integrated Gradients (IG): The Core Idea}

Integrated Gradients~\cite{Sundararajan2017IntegratedGradients} is a technique
for finding out how much each input feature contributed to a model's output.
The key insight is to compare the \emph{actual} input against a neutral
\emph{baseline} (e.g.\ a blank image or a zero vector) and measure how the
output changes as you gradually ``turn on'' each feature from the baseline
to the real value.

Concretely: imagine sliding each input feature $x_i$ from baseline $x'_i$ to
its actual value $x_i$, one tiny step at a time, while watching how the output
changes. The attribution score for feature $i$ is:
\begin{equation}
    \mathrm{IG}_i(x)
    \;=\;
    (x_i - x'_i)
    \int_0^1
    \frac{\partial F\!\bigl(x' + \alpha(x - x')\bigr)}{\partial x_i}
    \,d\alpha,
    \label{eq:ig}
\end{equation}
where $F$ is the model's output score and $\alpha$ is how far along the path
from baseline to real input we are. In practice this integral is approximated
by taking $N$ evenly spaced steps.

IG satisfies an important property called \emph{completeness}: the sum of all
attribution scores always equals $F(\text{real input}) - F(\text{baseline})$.
This means the attributions fully account for the change in output---nothing is
left unexplained. This is used as a built-in sanity check.

\subsection{Blur Integrated Gradients (BlurIG): Better Baselines for Images}

Standard IG uses a \emph{black image} (all zeros) as the baseline. For images,
this is problematic because a completely black image is very unnatural and can
introduce artifacts in the gradients.

Blur Integrated Gradients~\cite{Xu2020BlurIG} uses a smarter path: instead of
going from black to the real image, it goes from a \emph{maximally blurred}
version of the image (all fine details washed out, only rough colour blobs
remain) to the sharp original. This makes more intuitive sense: the blurred
image keeps the global colour and layout but removes the specific object details
we want to attribute.

The ``blurring'' is done by applying a Gaussian blur, which is a standard
image-smoothing operation controlled by a parameter $\sigma$ (higher $\sigma$
= more blurring). The attribution path sweeps $\sigma$ from a large value
(very blurry) down to zero (original sharp image):

\begin{equation}
    \mathrm{BlurIG}(E)
    \;\approx\;
    \sum_{k=0}^{K-1}
    \underbrace{\frac{\partial F(E_{\sigma_k})}{\partial E_{\sigma_k}}}_{\text{gradient at blur step }k}
    \;\cdot\;
    \underbrace{(E_{\sigma_{k+1}} - E_{\sigma_k})}_{\text{change in embedding}},
    \label{eq:blurig}
\end{equation}
where $E_{\sigma_k}$ is the image embedding blurred at level $\sigma_k$.

\paragraph{Why we work in embedding space, not pixel space.}
Instead of blurring raw pixels, we blur the output of the \textbf{SigLIP}
vision encoder~\cite{Zhai2023SigLIP}. SigLIP converts a $384 \times 384$ image
into a $27 \times 27$ grid of 1152-dimensional vectors (called ``patch
embeddings''). Each cell in the grid represents a $14 \times 14$ pixel patch.
We blur in this embedding space because: (1) it is the actual input to RDT,
so gradients flow cleanly; and (2) 27$\times$27 is much cheaper to compute
than full-resolution pixel attribution.

\subsection{Language Encoding with T5}

The text instruction (e.g.\ ``pick up a red cube'') is first converted to
numbers by a \textbf{T5}~\cite{Raffel2020T5} language model. T5 breaks the
sentence into small pieces called ``tokens'' (roughly one per word), and
produces a fixed-length sequence of vectors---one per token---that encode
the meaning of each word in context.

The pre-encoded token vectors are downloaded from HuggingFace as \texttt{.pt}
files, so the full T5 model does not need to run at inference time.

\subsection{Action Chunking}

Rather than re-planning at every single step, RDT predicts a chunk of 64
future actions at once and the robot executes the first 16 before requesting
a new prediction. This is called \emph{action chunking}~\cite{Zhao2023ACT}.
The advantage is that the robot commits to a locally coherent short-term plan
instead of being jittery from step-by-step re-planning.

\section{System Setup}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{The Robot Arm}

We use a Franka Panda, a popular research robot arm with 7 rotational joints
plus a two-finger gripper. We treat the gripper as a single number (average
of both finger positions), giving 8 values total. We give these joints
descriptive names for the visualisations:

\begin{center}
\textbf{base rot} $\to$ \textbf{shoulder} $\to$ \textbf{upper arm} $\to$
\textbf{elbow} $\to$ \textbf{forearm rot} $\to$ \textbf{wrist pitch} $\to$
\textbf{wrist rot} $\to$ \textbf{gripper}
\end{center}

In all heatmaps, joints are displayed with the gripper at the \emph{top}
and the base at the \emph{bottom}, matching the way the arm looks hanging
from a table mount.

\subsection{The Six-Camera Slot System}

RDT was designed for robots with multiple cameras (left wrist, right wrist, and
an external overview camera, each at two consecutive timesteps). ManiSkill
only provides one external camera. For the missing slots we fill in a neutral
background image (the SigLIP model's own mean colour), so the model still
receives a valid-length input. The six slots are:
\[
[\underbrace{\mathrm{external}_{t-1}}_{\text{prev.\ frame}},\;
 \underbrace{\text{blank},\;\text{blank}}_{\text{wrist slots, prev.}},\;
 \underbrace{\mathrm{external}_{t}}_{\text{current frame}},\;
 \underbrace{\text{blank},\;\text{blank}}_{\text{wrist slots, curr.}}]
\]

\subsection{Normalisation}

All joint angles are rescaled to the range $[-1, 1]$ using statistics
collected from ManiSkill data:
\begin{equation}
    \hat{s}_i = \frac{s_i - s_i^{\min}}{s_i^{\max} - s_i^{\min}} \times 2 - 1.
\end{equation}
Predicted actions are scaled back to real physical units (radians for joints,
a $[-1, 1]$ open/close signal for the gripper) before being sent to the
simulator.

\section{Methods}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{Collecting Successful Episodes}

Before we can run attribution, we need a real successful episode to analyse.
The rollout procedure:
\begin{enumerate}
    \item Reset the environment with a random seed.
    \item At each step, encode the current and previous camera frames with
          SigLIP and feed them to RDT together with the current joint positions.
    \item RDT outputs 64 predicted actions. Execute the first 16 in the
          simulator, then re-plan (action chunking with $K=16$).
    \item If ManiSkill reports success, save 10 evenly-spaced frames from the
          episode plus the joint state at each frame.
    \item Repeat until enough successes are collected.
\end{enumerate}

\subsection{Feature-Level BlurIG: Highlighting Important Image Regions}

For each saved frame we compute a $27 \times 27$ attribution map showing which
image patches mattered most. The steps are:

\begin{enumerate}
    \item Encode the frame with SigLIP to get patch embeddings $E$.
    \item Define a score: the magnitude of the robot's predicted
          \emph{gripper} commands over the next 8 steps---
          $F(E) = \|\mathbf{a}_{1:8,\,\text{gripper}}\|_2$.
          A high score means the model is predicting an active gripper movement.
    \item Run the BlurIG path: blur $E$ at $K$ decreasing sigma values
          ($\sigma_0 > \sigma_1 > \cdots > \sigma_K = 0$), compute the
          gradient of $F$ at each blur level, and accumulate the product
          of gradient and embedding change (Equation~\ref{eq:blurig}).
    \item Sum the absolute attribution across the 1152 channels at each
          of the 729 patch positions and normalise to $[0, 1]$.
    \item Upsample the $27 \times 27$ map to full image resolution and
          overlay it with a warm colour map (\texttt{inferno}: dark purple =
          low, bright yellow = high).
\end{enumerate}

\paragraph{Per-joint version.}
Instead of one score for the gripper, we compute a separate score for
each of the 8 joints ($s_j = \|\mathbf{a}_{:,j}\|$) and take a separate
gradient for each joint. This lets us say ``joint $j$ was paying attention
to patch region $r$'' and produces both per-joint scalar totals and
a joint-resolved spatial map---all in one pass through the blur schedule
(rather than running the blur twice, which would give inconsistent results).

\subsection{Word $\times$ Joint IG: Which Words Drive Which Joints}

For language attribution we apply standard IG (Equation~\ref{eq:ig}) over
the T5 token embedding vectors. The baseline is the all-zero vector
(``empty language'') and the target is the actual task embedding.

For each of the 15 interpolation steps, we compute a per-joint score
and take its gradient with respect to the language embedding. The final
result is an $8 \times W$ matrix where rows are joints and columns
are word groups, and each cell answers: ``how much did this word
influence this joint?''

Two views are shown:
\begin{itemize}
    \item \textbf{Row-normalised}: for each joint, which words does it
          pay the most attention to (relative to its own maximum)?
    \item \textbf{Global-normalised}: how do attributions compare
          across all joints and all words simultaneously?
\end{itemize}

\subsection{Fixing the Random Noise Confound}
\label{sec:seed}

RDT's denoising process starts from a random noise sample. If we call
the denoising function 20 times (once per BlurIG step) with a fresh
random noise each time, then differences in the score between steps will
partly reflect the randomness of the noise---not just the blur we changed.
This would corrupt the attribution.

\textbf{Fix:} we re-set the random number generator to the same fixed seed
immediately before every denoising call. This makes all 20 calls start from
\emph{identical} noise, so the only thing that differs between steps is the
blur level we are testing. Score differences then purely reflect the effect
of the blur, giving clean gradients.

\subsection{Fixing the Padding Token Bug}
\label{sec:pad}

\paragraph{The problem.}
T5 always outputs a fixed number of tokens (say 64), regardless of how
long the sentence is. A short instruction like ``stack the cubes'' might
only use 8 real tokens; the remaining 56 positions are \emph{padding}---
placeholder slots with no linguistic meaning.

The pre-encoded \texttt{.pt} files downloaded from HuggingFace did not
include an attention mask (a list of which positions are real vs.\ padding).
Both scripts defaulted to treating \emph{all} positions as real, so
RDT attended to the padding positions just like actual words.

Crucially, T5's padding positions are \emph{not} zero-valued in the output---
the transformer still computes non-trivial numbers there due to how self-attention
works across the full sequence. RDT picking up on these non-zero padding values
caused the word$\times$joint heatmap to show bright columns labelled
\texttt{[pad]}, as if meaningless filler tokens were strongly driving the
robot's joints.

\paragraph{The fix.}
We use the T5 tokeniser (a small, fast component) to count how many
real tokens the task description actually has:
\begin{equation}
    n_{\mathrm{real}} = |\texttt{tokenise}(\text{task description})|.
\end{equation}
We then build an attention mask that is 1 for positions $0 \ldots n_{\mathrm{real}}-1$
and 0 for everything after. After this fix, padding columns appear uniformly
dark in all heatmaps.

\section{Results: StackCube-v1}
% ─────────────────────────────────────────────────────────────────────────────

We present results from a successful StackCube-v1 episode (episode 8).
The task is: pick up the red cube and place it on top of the green cube.

\subsection{BlurIG Image Overlays}

Figure~\ref{fig:blurig_overlay} (top row of images) shows the 10 episode
frames with BlurIG attribution overlaid. Key observations:

\begin{itemize}
    \item \textbf{Start frame}: attribution is spread across the scene---the
          model is scanning the table to locate both cubes and the arm position.
    \item \textbf{Mid-episode (steps 37--62)}: attribution concentrates
          strongly on the cubes and the arm/gripper region. This is when the
          robot is reaching for and grasping the red cube.
    \item \textbf{Near success (step 99 -- SUCCESS)}: attribution re-concentrates
          on the gripper and the stacking contact point, as the robot carefully
          lowers the red cube onto the green one.
\end{itemize}

\subsection{Mean Attribution Over Time}

The mean BlurIG attribution per frame (Figure~\ref{fig:temporal}) shows a
clear rise-and-fall pattern:
\begin{itemize}
    \item \textbf{Low at start} ($\approx 0.033$): the robot is in its initial
          position with no urgent action required.
    \item \textbf{Peak at step 49} ($\approx 0.070$): the robot is actively
          reaching and grasping, requiring precise visual feedback.
    \item \textbf{Drop at step 74} ($\approx 0.030$): the robot has already
          gripped the cube and is executing a pre-planned lift; less real-time
          visual guidance is needed.
    \item \textbf{Second peak at SUCCESS} ($\approx 0.067$): the robot is
          placing the cube, again relying heavily on visual feedback to align
          the cubes for stacking.
\end{itemize}

\subsection{Per-Joint Attribution Over Time}

The joint heatmap (Figure~\ref{fig:joints}) reveals which joints relied most
on visual information at each point:

\begin{itemize}
    \item \textbf{Gripper} (top row): consistently high throughout. Expected,
          since the gripper must track the cube to open and close at the right time.
    \item \textbf{Wrist rot \& wrist pitch}: high early (locating the cube),
          drops sharply around step 37 as the arm commits to an approach angle,
          then rises again near the placement.
    \item \textbf{Elbow \& shoulder}: show a pronounced peak at step 74, when
          the arm is repositioning for the stacking placement---these large
          joints need visual confirmation of the destination.
    \item \textbf{Upper arm} (bottom of the visual rows, shown in green):
          consistently the \emph{lowest} attribution of all joints throughout
          the episode. The upper arm does coarse gross positioning that is
          largely pre-planned and does not need fine visual updates.
\end{itemize}

\subsection{Word $\times$ Joint Attribution}

The word$\times$joint heatmap (Figure~\ref{fig:wxj}) uses the start frame
and shows which words in ``Pick up a red cube and stack it on top of a
green cube'' influenced each joint.

After the padding fix, the pad columns \texttt{[p20]} onward are all dark---
the fix worked. Key observations from the global-normalised view:

\begin{itemize}
    \item \textbf{Wrist rot} has the highest overall attribution scores:
          0.61 on ``pick'', 0.78 on ``up'', 0.92 on ``green''/``cube'',
          and 0.58 on the final ``cube''. The wrist rotation joint is the most
          language-sensitive joint---it must orient the gripper in the right
          direction, so words describing the target object strongly influence it.
    \item \textbf{``green''} and \textbf{``cube''} (second occurrence) receive
          the highest attribution across all joints. The second cube mentioned
          in the sentence is the \emph{destination}---where the robot must
          place the red cube. The model clearly learned that the second object
          word specifies the goal, not just the object to pick.
    \item \textbf{Upper arm, elbow, shoulder}: very low attribution on all words
          in the global view, confirming that proximal joints are driven more
          by the spatial/visual plan than by linguistic content.
\end{itemize}

\subsection{Software}

\begin{itemize}
    \item \textbf{Simulation}: ManiSkill3 with SAPIEN physics; headless

    \item \textbf{Models}: RDT-1B and SigLIP loaded from HuggingFace.
    \item \textbf{Deep learning}: PyTorch with bfloat16 precision on GPU.
    \item \textbf{Plots}: Matplotlib; attribution maps use the

          \texttt{rdgbrandon/rdt-igtesting} on GitHub.

\subsection{Stub Trick for Unused Dependencies}

RDT's source code imports \texttt{wandb}, \texttt{deepspeed}, and
\texttt{flash\_attn} at the top level, but these are only needed for
\emph{training}, not inference. Rather than installing these large packages,
we create lightweight ``stub'' objects that pretend to be those modules
and return harmless placeholder values whenever they are called.
This lets the model load without those dependencies installed.

\section{Discussion}
% ─────────────────────────────────────────────────────────────────────────────

\subsection{What the Results Tell Us}

The BlurIG results make intuitive sense: the robot attends most strongly to
the objects it needs to interact with and to its own gripper, with attention
peaking at the moments that require the most precise visual feedback (approach,
grasp, and placement). This is encouraging evidence that RDT-1B has learned
genuine task-relevant visual attention rather than spurious correlations.

The word$\times$joint results show that the wrist joints are the most
language-sensitive, which is physically sensible: the wrist controls the
orientation of the gripper, and the correct orientation depends on what
object the robot is supposed to pick up and where it is supposed to place it.
The clear distinction between ``red cube'' (pick target) and ``green cube''
(place target) shows the model distinguishes object identity from goal identity
from language.

\subsection{Limitations and What We Fixed}

\paragraph{Precision tasks.}
PegInsertionSide and PlugCharger tasks had much lower success rates than
PickCube or StackCube. We identified that one cause was a stride-4
subsampling of the predicted action trajectory (\texttt{acts[::4]}):
this skipped three out of every four predicted positions, making the
robot's movement jerky and imprecise---which is tolerable for picking a
large cube but causes failure when trying to insert a small peg into a
precise slot. The fix (use the first 16 \emph{consecutive} predicted
actions instead) produces smoother motion.

\paragraph{Padding token attribution.}
Before fixing the attention mask, up to 70\% of the word$\times$joint
heatmap was occupied by padding columns that appeared to have high
attribution. After the fix, attribution correctly concentrates on the
actual task words.

\paragraph{Camera view mismatch (discovered).}
We briefly tried pulling frames from the robot's internal sensor camera
(\texttt{obs['sensor\_data']}) instead of the external render camera
(\texttt{env.render()}). This broke \emph{all} tasks including PickCube,
confirming that the model had been fine-tuned on external render views.
This change was reverted.

% ─────────────────────────────────────────────────────────────────────────────
\section{Conclusion}
% ─────────────────────────────────────────────────────────────────────────────

This project applied Blur Integrated Gradients and Integrated Gradients to
the RDT-1B robot policy to answer a simple but important question: what is
the robot ``looking at'' and ``listening to'' when it moves? The results show
that the policy focuses on task-relevant visual regions (objects and gripper)
and that different joints are driven by different words in the instruction.
Along the way, two important bugs were found and fixed---a noise confound in
the gradient computation and a padding token attribution artifact---improving
the reliability of the results. The pipeline is fully open-source and
produces five complementary visualisations per episode that could be used to
debug and improve robot policies on other tasks.
